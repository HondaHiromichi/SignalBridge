# Flair 連携 確認手順書 (Phase 1 / Phase 2)

> SignalBridge の OSC テスターを使って実機 Flair と疎通するときの確認項目と,
> 状態別の切り分け (決定木) をまとめたもの. ツールの役割は [WORKLOG.md](WORKLOG.md) を参照.

## 前提

- ムーブ/ジョブ起動の OSC 仕様 (アドレス文字列・引数の型/個数・受信側設定) は **未確定**
  (メーカー/購入元の回答待ち). 本ツールは「仕様が来たら UI に値を入れるだけ」で繋がる設計.
- 本書の具体値 (アドレス例など) は **未確認の例示**. 出所はメーカー回答 /
  `Flair v7.4 Operator's Manual` / `確認質問リスト.pdf` (いずれもリポジトリ外).
- ネットワーク図: `[Unity 機] --(OSC / UDP)--> [Flair PC] --(EtherCAT 等)--> [アーム]`

---

## A. 事前に確定する項目

### 送信 (Phase 1) に必要
- [ ] Flair PC の **IP アドレス**
- [ ] Flair の **OSC 受信が有効か / 受信ポート番号**
- [ ] ムーブ/ジョブ起動の **OSC アドレス文字列** (例: `/flair/job` `/move` 等 = 未確認)
- [ ] **引数の有無・型・個数・意味** (例: ジョブ番号 int? プリセット名 string? なし?)
- [ ] ネットワーク: Unity 機と Flair が **同一サブネット**か / **UDP がファイアウォールを通る**か

### 受信 (Phase 2) に必要
- [ ] Flair の **Data Output (XYZ 等) 機能の有無**と有効化方法
- [ ] その **送信先 IP / ポート** (= Unity の受信ポートに合わせる)
- [ ] 出力の **アドレス / 型 / レート**

---

## B. Phase 1 (送信) 手順

1. ネットワーク疎通: OS で `ping <FlairのIP>` が通るか
2. 宛先 IP = Flair, 宛先 Port = Flair の受信ポート を設定
3. 確定したアドレス/引数を入れて **Send**
   (ログの `SEND ... (N bytes) hex=...` で送出を確認)
4. **アームが動くか** + **Flair 側の受信表示/ログ** を確認

### 状態別の切り分け (送ったのに動かない)

| 状態 | 疑う所 -> 試すこと |
|------|--------------------|
| `SEND` ログが出ない / 例外 | 入力ミス (ポート不正等). ログの `ERROR(send)` を読む |
| `SEND` は出るが Flair 無反応 | (1) ネットワーク: ping 通るか / 同一サブネットか / UDP が FW で遮断されてないか |
| ネットワーク OK でも無反応 | (2) ポート: 宛先 Port と Flair 受信ポートが一致か / Flair で OSC 受信が有効か |
| ポートも OK でも無反応 | (3) アドレス文字列: 大小文字・スラッシュ・末尾まで完全一致か. 候補をプリセットで総当たり |
| アドレスらしいが反応薄い | (4) 引数: 型 (i/f/s...)・個数・順序が違う可能性. `iif` `i` `s` 等をプリセットで切替 |
| Flair が受けてるが誤動作/エラー | 引数の意味・順序・エンディアン. 値を変えて挙動の対応を観察 |
| エンコーダで表現できない並びが要る | Raw 送信: 正しいはずの hex を直接送ってバイト単位で合わせる |

> 送信バイト (hex) がログに出ているのに動かない = ほぼ ネットワーク / ポート / アドレス /
> 引数 のどれか. 上から順に切り分けるのが速い.

---

## C. Phase 2 (受信) 手順

1. Flair 側で **Data Output (XYZ 等) を有効化**し, 送信先を Unity 機の IP・任意ポートに設定
2. Unity の **受信 Port をそのポートに合わせ, Listen を ON**
3. ログの `RECV from <ip:port> <address> <type> <値>` を観察

### 状態別の切り分け

| 状態 | 試すこと |
|------|----------|
| 何も `RECV` されない | Flair の Data Output が有効か / 送信先 IP・ポートが Unity と一致か / FW で UDP 受信が通るか / 受信 Port が他で使用中でないか |
| `RECV` は来るが内容が変 / `[未デコード]` 注記 | 寛容デコードの `hex=...` を見て型を逆算 (4 バイト境界・ビッグエンディアンで int/float か等). アドレス文字列も hex から確認 |
| 送信元が想定と違う | `from <ip:port>` で誰が送ってるか確認 (別機器/別ポートの混線を排除) |

---

## D. ツール機能の使い分け

- **送信 (Send)**: 仕様が分かっている時の本番投入. プリセットで候補を回す.
- **Raw 送信 (hex 直送)**: 仕様が読めない/特殊な時の総当たり・ビット単位検証 (最後の手段).
- **受信 (Listen)**: Flair が何を返すかの観察. 仕様不明でも hex から逆算できるのが寛容デコードの狙い.
- **プリセット**: 送信設定の組を保存/呼出し, 現場で素早く切り替える.
- **ログ (フィルタ/Clear/色分け)**: 送受信・例外の観察面. SEND / RECV(緑) / ERROR(赤).
